Using Git with Subversion

The bridge to Subversion in Git is provided by the in-built git-svn

.

Supported Features

  • Cloning a Subversion server repository to local Git
  • Fetching new revisions from Subversion and displaying them as if they are a remote branch
  • Progressive pull of new revisions from remote Subversion into local Git commits
  • Progressive push of new local Git commits to remote Subversion server revisions
  • Working within existing Subversion branches
  • Creating new Subversion branches & tags

Unsupported Features

The following features are not supported by git-svn:

  • Merging Subversion branches

Cloning Subversion To Git

When you clone a Subversion URL, you have a choice - you can either clone a single branch of the repository (or the trunk), represented in Subversion by unique URLs, or you can clone from the root of the repository, and if the repository is in the standard Subversion layout (branches, trunk and tags folders in the root), then the conversion will pick up the branches and tags and convert them appropriately.

Pushing and Pulling

Subversion only allows a linear history, compared to Git which allows parallel development (resolved by rebasing or merging). Therefore when pulling and pushing changesets from/to Subversion, the history has to be 'flattened' in each branch, which is done via rebase.

Only the active branch is pushed or pulled to/from Subversion, and it needs to be a branch which was checked out from a Subversion branch (which is like a remote branch). In practice, git-svn looks back in the history to the latest commit which has a Subversion revision ID on it (displayed in the commit message), and re-applies your commits against Subversion from there.

In git-svn speak, SourceTree maps 'Push' to 'git svn dcommit' when you select the special "Subversion" remote. We do this to retain the usual workflow you're used to.

Merging is not Supported

The git merge information simply cannot be mapped back to Subversion accurately, so in practice you are limited to using Cherry Pick to move changes between branches (or using svn merge elsewhere). You can in theory merge from local Git branches to Subversion branches, you have to be extremely careful not to allow any Subversion commits to become part of a different branch, otherwise chaos will ensue. In practice it's best just to avoid it, and SourceTree warns you to this effect.

git-svn also does not pick up merge information from merges that have occurred in Subversion already, so the log view will not show any merges, only branches.

Migrating away from Subversion

One of the side benefits of using the Subversion interoperability is that effectively you've converted your repository to Git just by cloning it. If you wanted to move away from Subversion later, you can use your local Git repository as a base. Just remember that merge information is not present so you may wish to just convert a single branch at an opportune moment in time.

If you no longer intend to use Subversion, delete the svn metadata which lives inside the .git/svn folder in your local repository, and manually delete the 'svn-remote' entry in .git/config. From that point on, the repository will act like a regular Git repository.
